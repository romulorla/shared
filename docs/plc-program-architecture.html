Platform

  • PLC: Siemens S7-1516 (S7-1500 series)
  • Programming Environment: TIA Portal V19 Update 3, STEP 7 Professional
  • Languages Used: LAD (Ladder Diagram), GRAPH (Sequential Function Chart), SCL (Structured Control Language), STL (Statement List), DB (Data Block)

Module Pattern

Every module in the RadialPress PLC program follows a standardized block structure:

  • [XX]Main (OB) – Module entry point (individual OB per module)
    • [XX]Call (FB) – Orchestrator: calls all sub-blocks in order
      • [XX]Simulation (FB) – Simulation mode: forces I/O signals for testing
      • [XX]EquipmentInput (FB) – Maps physical inputs (sensors → variables)
      • [XX]Core (FB) – Core logic: mode management, state coordination
        • [XX]Alarms (FB) – Alarm/fault detection via Program_Alarm
        • [XX]Interlocks (FB) – Safety interlocks via LInterlock8Block-AND
        • [XX]Buttons (FB) – HMI button handling and manual commands
        • [XX]StateMachine (FB) – GRAPH sequencer: automatic cycle steps
        • [XX]Control (FB) – Control logic: actuator commands, setpoints
          • Components/ – Module-specific component blocks
            • [XX]ComponentA (FB)
            • [XX]ComponentB (FB)
      • [XX]EquipmentOutput (FB) – Maps variables → physical actuator outputs

Where [XX] is the 2-letter module prefix (e.g., WT for Work Table, GB for Gearbox).

Execution Order Within a Module

The call order within each module's Call block is significant:

  1. Simulation – First, so simulation can override I/O before processing
  2. EquipmentInput – Read physical inputs (or simulated ones)
  3. Core – Process logic with current inputs (calls Alarms → Interlocks → Buttons → StateMachine → Control)
  4. EquipmentOutput – Write computed outputs to physical actuators

Variations From the Standard Pattern

Not all modules implement every block:

  • Module 1300 (Hydraulics): No StateMachine, Buttons, or Interlocks (operates as a utility/service module)
  • Module 900 (Water Sprinkling): No Alarms or Simulation
  • Module 1000 (Pallet Centering): No Alarms or Simulation
  • Module 1400 (Skip Hoist): No Alarms or Simulation. Still not completely developed.

Data Block Pattern

Each module has a main instance DB named after the module (e.g., WorkTable, Gearbox, PressShaft). This DB serves as the instance DB for the module's Call FB and contains all the module's internal state.

Data Organization Within Module DBs

Module DBs typically contain sections for:

  • Recipe – Parameters for the automatic sequence (setpoints, speeds, positions, timers) that change per product/pipe type
  • MachineData – Hardware configuration parameters (sensor types, calibration, physical limits) that are set during commissioning
  • Status – Runtime status variables (current state, positions, flags)
  • Interface – Cross-module communication variables

Recipe vs MachineData Pattern

  • Recipe data defines what the machine should do for a given product (e.g., target positions, speeds, fill times). Operators may change these per pipe type.
  • MachineData defines how the hardware is configured (e.g., sensor scaling, encoder resolution, mechanical limits). These are set during commissioning and rarely change.

Programming Patterns

State Machine Pattern (GRAPH)

  • Each module's StateMachine block is a GRAPH sequencer defining the automatic cycle
  • Steps represent physical states/actions (e.g., "Move down", "Start feeder", "Compacting")
  • Each step has actions (outputs set while the step is active)
  • Transitions between steps contain conditions (sensor feedback, timers, interlocks, cross-module signals)
  • The StateMachineDB is the instance DB that stores the GRAPH block's state
  • Some modules have branching/alternative paths (e.g., repetition cycles in Gearbox and Conveyor Belt)
  • Dual sequences exist in Water Sprinkling (two parallel GRAPH sequences)

Interlock Pattern

  • Interlocks are checked in the Interlocks block using library block LInterlock8Block-AND
  • Each interlock group combines up to 8 conditions with AND logic
  • Conditions include: safety signals, position feedback, cross-module states, mode flags
  • Interlock outputs are passed to Control/Component blocks to enable/disable actuator movement
  • Common interlock groups: "Common" (applies to all movements), directional groups (e.g., "Move Up", "Move Down")

Equipment I/O Pattern

  • EquipmentInput maps physical sensor signals to module-internal variables
  • EquipmentOutput maps internal command variables to physical actuator outputs
  • This abstraction layer enables simulation: in simulation mode, Simulation block forces signals before EquipmentInput reads them
  • Physical I/O addresses appear as %I (inputs) and %Q (outputs) in these blocks only
  • Library blocks used: LDigitalInput for debounced digital inputs, L_AnalogScaling for analog signal scaling

Alarm Pattern

  • Alarms use the Program_Alarm instruction from the Siemens library
  • Each alarm has a unique ID within the module (e.g., 101–106 for Work Table, 601–603 for Gearbox)
  • Alarm acknowledgment uses the Ack_Alarms library block
  • Alarm conditions are typically: false positions (actuator didn't reach expected position in time), circuit breaker trips, sensor faults, communication errors

Safety Integration

  • Safety outputs come from SFOutputDB (safety function output data block)
  • Global safety signals are accessed via Global.safety.* paths
  • Safety interlocks are incorporated into the interlock groups alongside operational conditions

HMI Button Pattern (Buttons Block)

  • Dual joystick support: left and right joysticks with selectable function assignment per side
  • Button actions for manual mode: directional controls (up/down, forward/backward), start/stop
  • Mode selection: Automatic, Manual, Setup modes
  • Buttons block writes to control flags that are consumed by the Control block

Cross-Module Communication

Modules never call each other directly. All inter-module coordination flows through two global data blocks: Interface (DB 4) for module-to-module status and commands, and Global (DB 1) for machine-wide state.

Interface Block (DB 4)

The Interface data block is the central communication hub. Each module has its own namespace within it:

  • Interface (DB 4)
    • workTable (typeWorkTableInterface)
    • cageCentering (typeCageCenteringInterface)
    • turnTable (typeTurnTableInterface)
    • bellCompactor (typeBellCompactorInterface)
    • conveyorBelt (typeConveyorBeltInterface)
    • gearbox (typeGearboxInterface)
    • pressShaft (typePressShaftInterface)
    • distributorShaft (typeDistributorInterface)
    • waterSprinkling (typeWaterSprinklingInterface)
    • palletCentering (typePalletCenteringInterface)
    • palletInserting (typePalletInsertingInterface)
    • hydraulic (typeHydraulicInterface)
    • skipHoist (typeSkipHoistInterface)
    • concreteOrder (typeConcreteOrderInterface)

Each module's interface section follows a consistent pattern with two areas:

  • status — The module writes its current state here (positions, running flags, alarm flags, temperatures). Other modules and the global sequence read these fields.
  • command — The global sequence or other modules write commands here (step triggers, speed setpoints, manual controls). The module reads them as inputs.

The Interface block is passed as an InOut parameter through the call chain: [XX]Call receives it, passes it to [XX]Core, which passes it down to [XX]Alarms, [XX]Control, [XX]StateMachine, etc. Each block reads or writes the fields it needs.

Communication Flow Example

A typical production cycle step works like this:

  1. GlobalSequence writes Interface.gearbox.moveToBottom = TRUE
  2. Gearbox reads this command, starts lowering the shaft
  3. Gearbox writes Interface.gearbox.status.cycleDone = TRUE
  4. GlobalSequence reads cycleDone, transitions to the next step

Global Block (DB 1)

The Global data block holds machine-wide state that every module reads. Unlike Interface, which is per-module, Global contains state that applies to the entire machine:

  • Global (DB 1)
    • simulation (Bool) — Simulation mode flag
    • mode — Operating mode (Retain)
      • automatic (Bool) — Automatic production mode (default: TRUE)
      • manual (Bool) — Manual/jog mode
      • setup (Bool) — Setup/commissioning mode
      • cleaning (Bool) — Cleaning mode
    • commands — Operator commands (Retain)
      • powerOn (Bool) — Master power enable
      • automaticStart (Bool) — Start automatic cycle
      • automaticStop (Bool) — Stop automatic cycle
      • reset (Bool) — Alarm reset / acknowledge
      • resetSequence (Bool) — Reset sequence counter
      • initialPosition (Bool) — Go to home position
      • regulationOff (Bool) — Disable pressure regulation
      • requestToStop (Bool) — Operator requests stop
      • allowRequestedStop (Bool) — Machine reached safe position, can stop
      • lampTest (Bool) — Test indicator lamps
    • safety (typeGlobalSafety) — Safety signals (access doors, e-stop)
    • joystick — Four joystick inputs (Retain)
      • topLeft / topRight / bottomLeft / bottomRight
      • each: up, down, left, right (Bool)
    • status
      • powerOn (Bool) — Power is active
      • automaticStopped (Bool) — Automatic mode halted
      • atInitialPosition (Bool) — Machine at home position
      • startActive (Bool) — Start LED indicator
      • startCondition (Bool) — Start conditions met
    • withHandlingLine (Bool) — Handling line configuration flag

Modules access these as Global.mode.automatic, Global.commands.reset, Global.safety.accessDoorsOK, Global.simulation, etc.

Global Blocks Hierarchy

The Global layer has its own block structure that runs alongside the per-module OBs:

  • GlobalCall (FB 11, DB "GlobalCall_DB")
    • GlobalControlPanel (FB 13) — Button/LED handling, mode selection
    • GlobalSignals (FB 14) — Access door requests, safety alarm routing
    • GlobalAlarms (FB 16) — 9 system-wide Program_Alarm instances
    • GlobalSequence (GRAPH 10) — Main production sequence state machine
    • GlobalInitialPosition (GRAPH 12) — Startup homing sequence

Supporting blocks:

  • StateMachines (DB 6) — Instance data for both GRAPH sequences
  • Production (20) — Production cycle management (ProductionMain, ProductionCore)
  • Maintenance (30) — Maintenance mode logic (MaintenanceMain, MaintenanceCore)
  • Help functions (40) — Access request handling (HFAccessRequest), safety alarms (HFSafetyAlarms)

GlobalSequence is the master orchestrator of the automatic production cycle. It writes commands to each module's Interface section (e.g., Interface.gearbox.moveToBottom = TRUE) and waits for status feedback (e.g., Interface.gearbox.status.cycleDone). GlobalInitialPosition performs the same coordination pattern but for the startup homing sequence.

Shared Resources

  • SEW Frequency Drive: Turn Table (300) and Conveyor Belt (500) share a single SEW MoviC Advanced frequency drive. Only one can operate at a time — mutual exclusion enforced through interlocks.
  • Hydraulic Power Unit: Module 1300 provides central hydraulic pressure to Work Table, Gearbox, Bell End Compactor, Press Shaft, Distributor Shaft, and Pallet Centering. All hydraulic consumers check Interface.hydraulic.status.drivesOn as an interlock before commanding movement.
