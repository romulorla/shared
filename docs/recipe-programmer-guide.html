Recipe System — Programmer Guide

PLC Architecture • Data Flow • Storage Optimization • FC Operations

Internal technical reference for the recipe management system. Covers the three data areas, two function blocks, the flattening optimization, all FC operations, and how to extend the system with new parameters. Up to 100 recipes stored in retentive PLC memory.

System Architecture

The system is built around three data areas and two function blocks, with HMI lookup arrays for fast list display.

Data Blocks
Data Block
Recipe RETAIN

Active recipe used by the machine during production. Type: typeRecipe. This is what the PLC actually executes against — all module parameters come from here.

Data Block
RecipeHMI NON-RETAIN

Working copy displayed and edited on the HMI panel. Type: typeRecipe. Temporary buffer — unsaved changes are lost on power loss. Operators interact with this copy.

Data Block
RecipeStorage RETAIN

Persistent storage for all saved recipes. Type: Array[1..100] of typeRecipeFlat. Uses the flattened UDT for 97% memory reduction. Survives power cycles.

Function Blocks
FC
RecipeHandling

Processes all HMI button commands: create, edit, delete, read, save, save as, sync, download, and the async save-from-module operation.

FC
FC_RecipeStorageMap

Low-level mapper — explicit field-by-field assignment between RecipeHMI.data and RecipeStorage.recipe[N]. Handles both save (flatten) and load (expand) directions.

HMI Lookup Arrays

These arrays mirror recipe header info so the HMI list can display names, types, and dimensions without reading the full storage.

ArrayContent
#ioRecipeNamesProduct names
#ioRecipeTypesProduct types
#ioRecipeDiametersNominal diameters
#ioRecipeHeightsProduct heights
#ioRecipeWallThicknessesWall thicknesses

Storage Optimization — Flattened UDT

The full typeRecipe UDT is ~47.7 KB per recipe (WString descriptors, metadata, HMI-only fields). Storing 100 copies would require ~4.7 MB of retentive memory — exceeding most S7-1500 CPUs. The flattened typeRecipeFlat stores only recipe-dependent fields.

RecipeHandling FC — Operations
Input validation: Prevents selectedRecipe from being 0 since recipes are 1-indexed. Automatically corrects to 1. All button signals are expected to be single-scan pulses.
Create single-scan
#interface.buttons.create

Writes #interface.newRecipeValues into #ioRecipeStorage at the specified ID. Updates all HMI lookup arrays so the list reflects the new entry immediately.

Edit single-scan
#interface.buttons.edit

Case A: Selected recipe matches loaded RecipeHMI → updates HMI working copy directly. Case B: Different recipe → updates storage only. Both cases update lookup arrays.

Delete single-scan
#interface.buttons.delete

Clears lookup arrays (empty strings, zeroes), overwrites storage slot with empty template (preserving program number), resets RecipeHMI.

Read single-scan
#interface.buttons.read

Sets #storageIndex and #loadFromStorage := TRUE. Actual copy performed by FC_RecipeStorageMap later in the same scan cycle.

Save single-scan
#interface.buttons.save

Sets #storageIndex (from RecipeHMI's program number) and #saveToStorage := TRUE. Copy performed by FC_RecipeStorageMap in the same scan.

Save As single-scan
#interface.buttons.saveAs

Copies entire storage record from source slot to target slot. Overwrites program number and name with user values. Updates lookup arrays for new slot.

Download single-scan
#interface.buttons.download

One-shot structure copy: #ioRecipe := #ioRecipeHMI. Machine immediately begins using the new parameters.

Sync single-scan
#interface.buttons.sync

Copies active PLC recipe into RecipeHMI. Identity fields (program number, name, type, dimensions) are backed up before copy and restored after.

Async Operation — Save From Module

The only multi-scan operation in the system. Spans two PLC scans, using the #asyncOperationRunning flag to latch across cycles.

Why multi-scan? When RecipeHMI holds a different recipe than the active PLC recipe, the system must first load the correct recipe from storage (Scan N) before it can sync runtime values and save (Scan N+1). The #asyncOperationRunning flag latches the operation across cycles.
Adding a New Recipe Parameter

Step-by-step guide for extending the recipe system when a new parameter is added to a machine module.

  1. Add field to typeRecipeFlat UDT
    Follow the naming pattern: {moduleName}_{switch|setpoint}_{fieldName}
    Example: extruder_setpoint_temperature
  2. Add a Save line in FC_RecipeStorageMap
    Map from the full recipe to the flat storage:
    RecipeHMI.data.xxx → RecipeStorage[N].xxx
  3. Add a Load line in FC_RecipeStorageMap
    Map from flat storage back to the full recipe:
    RecipeStorage[N].xxx → RecipeHMI.data.xxx
  4. If header/identity field: update lookup arrays
    Update the Create, Edit, Delete, and Save As sections in RecipeHandling to maintain the corresponding HMI lookup array.
⚠ Don't forget both directions! Every field needs a Save line (flatten: HMI → Storage) and a Load line (expand: Storage → HMI) in FC_RecipeStorageMap. Missing one direction means the parameter won't round-trip correctly.
Key Considerations
🔒 Retentive Memory
RecipeStorage is RETAIN. Recipes survive power cycles. Recipe (active) is also RETAIN.
⚡ RecipeHMI is Volatile
RecipeHMI is NON-RETAIN — it's a temporary working buffer. Unsaved changes are lost on power loss.
🔢 1-Based Indexing
Recipe index is 1-based. Slot 0 is not used. Input validation automatically corrects selectedRecipe = 0 to 1.
📡 Pulse-Based Buttons
Button signals must be single-scan pulses. Each operation runs once per press. The HMI should send single-scan triggers.
⏱️ Single-Scan Operations
All operations complete within one PLC scan cycle — except Save From Module, which spans two scans using the async flag.
🗺️ FC_RecipeStorageMap
Explicit 1:1 field assignment across 14 modules: 7 header + 6 alternating + 64 switches + 172 setpoints. Index validated (1..100).
