Recipe System — Programmer Guide

Siemens S7-1500 • PLC Architecture & Data Flow

The recipe system manages product recipes from the HMI panel — create, edit, store, load, and transfer to the active PLC. Each recipe defines product dimensions and all machine module parameters (setpoints & switches) needed to produce a specific pipe.

Three Data Blocks hold recipe data at different lifecycle stages. Two Function Blocks handle all operations.

DB Recipe
Type: typeRecipe
Retentive: Yes

Active recipe used by the machine during production. Downloaded from HMI, consumed by all machine modules.
DB RecipeHMI
Type: typeRecipe
Retentive: No

Working copy displayed and edited on the HMI panel. Temporary buffer — unsaved changes lost on power loss.
DB RecipeStorage
Type: Array[1..100] of typeRecipeFlat
Retentive: Yes

Persistent storage for all 100 recipe slots. Uses flattened type for 37× memory savings.
managed by
FC RecipeHandling
Processes all HMI button commands: create, edit, delete, read, save, save as, sync, download, and save from module. Orchestrates data flow between the three DBs.
FC FC_RecipeStorageMap
Low-level field mapper. Copies individual fields between RecipeHMI (full UDT) and RecipeStorage (flat). Handles both save (flatten) and load (expand) directions.
HMI Lookup Arrays — ioRecipeNames, ioRecipeTypes, ioRecipeDiameters, ioRecipeHeights, ioRecipeWallThicknesses mirror recipe headers so the HMI list can display names and dimensions without reading full storage records.

Save From Module is the only multi-scan operation. It first syncs runtime values from Recipe → RecipeHMI, then saves RecipeHMI → RecipeStorage across two PLC scan cycles using the asyncOperationRunning flag.
Storage Optimization

The full typeRecipe UDT is ~47.7 KB due to WString descriptors, metadata, and HMI-only fields. Storing 100 copies would require ~4.7 MB — exceeding most S7-1500 retentive memory. The flattened typeRecipeFlat stores only recipe-dependent fields.

✓ What IS Stored
  • 7 header fields (programNo, name, type, height, diameter, wall, concrete)
  • 6 alternating program fields
  • 64 component switches (Bool) across 14 modules
  • 172 setpoint values (Real) across 14 modules
✗ What is NOT Stored
  • WString descriptions (parameter labels)
  • Engineering units
  • Min / Max limits
  • Display settings (HMI layout metadata)
  • Static configuration (doesn't change between recipes)
Operations Detail
✚
Create SINGLE SCAN
Writes newRecipeValues directly into RecipeStorage[id]. Updates HMI lookup arrays immediately.
Trigger: buttons.create
✏️
Edit SINGLE SCAN
Case A: Selected = loaded in HMI → updates HMI copy directly.
Case B: Selected ≠ loaded → updates only storage array.
Both cases update HMI lookup arrays.
Trigger: buttons.edit
🗑️
Delete DESTRUCTIVE
Clears HMI lookup arrays, overwrites storage slot with empty template (preserving programNo), resets RecipeHMI.
Trigger: buttons.delete
📖
Read SINGLE SCAN
Sets storageIndex and loadFromStorage := TRUE. FC_RecipeStorageMap expands flat storage into RecipeHMI in the same scan.
Direction: Storage → RecipeHMI
💾
Save SINGLE SCAN
Sets storageIndex from RecipeHMI's programNo and saveToStorage := TRUE. FC_RecipeStorageMap flattens RecipeHMI into storage.
Direction: RecipeHMI → Storage
📋
Save As SINGLE SCAN
Copies entire storage record from source slot to target slot, then overwrites programNo and name with user values. Updates lookup arrays for new slot.
Direction: Storage[A] → Storage[B]
⬇️
Download SINGLE SCAN
One-shot structure copy: Recipe := RecipeHMI. Machine immediately begins using the new parameters.
Direction: RecipeHMI → Recipe (Active)
🔄
Sync SINGLE SCAN
Copies active Recipe into RecipeHMI while preserving identity fields (programNo, name, type, diameter, height, wall). Identity backed up before copy, restored after.
Direction: Recipe (Active) → RecipeHMI
📦
Save From Module MULTI-SCAN ASYNC
The only multi-scan operation. Syncs runtime PLC values and saves to storage across two scan cycles using the asyncOperationRunning latch flag.

Adding a New Parameter

When a new parameter is added to a machine module, follow these steps to integrate it into the recipe system:

  1. Add field to typeRecipeFlat UDT
    Naming pattern: {moduleName}_{switch|setpoint}_{fieldName}
  2. Add a Save line in FC_RecipeStorageMap
    RecipeHMI.data.xxx → RecipeStorage[N].xxx
  3. Add a Load line in FC_RecipeStorageMap
    RecipeStorage[N].xxx → RecipeHMI.data.xxx
  4. If header/identity field: also update HMI lookup arrays in RecipeHandling — in the Create, Edit, Delete, and Save As sections.
Key Considerations
💾 Retentive Memory
Recipe and RecipeStorage are RETAIN — they survive power cycles. RecipeHMI is NOT retentive. Unsaved changes are lost on power loss.
🔢 1-Based Indexing
Recipe slots are 1..100. Slot 0 is not used. Input validation in RecipeHandling automatically corrects selectedRecipe = 0 to 1.
⚡ Pulse Buttons
All HMI buttons must send single-scan pulses. Each operation triggers once per button press. The HMI should not hold signals high across multiple scans.
⏱️ Single vs Multi-Scan
Save From Module is the only multi-scan operation using asyncOperationRunning. All other operations (create, edit, delete, read, save, save as, download, sync) complete within a single PLC scan cycle.
