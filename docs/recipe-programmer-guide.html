Recipe System — Programmer Guide

PLC architecture, data flow, storage optimization, and function block reference

Author: Romulo Lange Date: March 2026 Platform: Siemens S7-1500 / TIA Portal Version: 1.0

1. Overview

The recipe system allows operators to create, edit, store, and load product recipes from the HMI panel. Each recipe defines the product dimensions and all machine module parameters (setpoints and switches) needed to produce a specific pipe.

The system supports up to 100 recipes stored in retentive PLC memory.

2. Architecture

The system is built around three data areas and two function blocks:

4. Why Flattened Storage?

The full typeRecipe UDT is ~47.7 KB per recipe due to WString descriptors, metadata, and HMI-only fields. Storing 100 full copies would require ~4.7 MB of retentive memory, which exceeds most S7-1500 CPUs.

The flattened typeRecipeFlat stores only the recipe-dependent fields:

typeRecipe (full UDT)
× 100 = 4.7 MB RETAIN
1.3 KB
typeRecipeFlat (optimized)
× 100 = 130 KB RETAIN

Static fields like descriptions, units, min/max limits, and display settings are not stored since they don't change between recipes.

5. RecipeHandling FC — Operations in Detail

5.1 Input Validation

Prevents selectedRecipe from being 0 since recipes are 1-indexed. Automatically corrects to 1.

5.2 Create

Triggered by: #interface.buttons.create

Writes all fields from #interface.newRecipeValues directly into #ioRecipeStorage at the index specified by the new recipe's ID. Also updates the HMI lookup arrays so the recipe list immediately reflects the new entry.

5.3 Edit

Triggered by: #interface.buttons.edit

Two cases are handled:

  • Case A: If the selected recipe is the one currently loaded in RecipeHMI (same program number), the HMI working copy is updated directly.
  • Case B: If the selected recipe is a different one, only the storage array is updated.

In both cases, the HMI lookup arrays are updated so the recipe list reflects the changes.

5.4 Delete

Triggered by: #interface.buttons.delete

Clears the selected recipe completely:

  1. Clears the HMI lookup arrays (empty strings, zeroes)
  2. Overwrites the storage slot with an empty template (preserving program number)
  3. Resets RecipeHMI to an empty template (preserving program number)

5.5 Read

Triggered by: #interface.buttons.read

Sets #storageIndex and #loadFromStorage := TRUE. The actual data copy is performed by FC_RecipeStorageMap later in the same scan cycle (Section 4 of the code).

5.6 Save

Triggered by: #interface.buttons.save

Sets #storageIndex (from RecipeHMI's program number) and #saveToStorage := TRUE. The actual data copy is performed by FC_RecipeStorageMap later in the same scan cycle.

5.7 Save As

Triggered by: #interface.buttons.saveAs

Copies the entire storage record from the source slot to a new target slot, then overwrites the program number and product name with user-supplied values. Updates the HMI lookup arrays for the new slot.

5.8 Download

Triggered by: #interface.buttons.download

One-shot structure copy: #ioRecipe := #ioRecipeHMI. The machine immediately begins using the new recipe parameters.

5.9 Sync

Triggered by: #interface.buttons.sync

Copies the active PLC recipe into RecipeHMI while preserving the recipe identity fields (program number, product name, product type, diameter, height, wall thickness). These fields are backed up before the copy and restored afterwards.

Use case: when the machine logic has modified runtime parameters and the operator wants to view or save those changes.

5.10 Save From Module

Triggered by: #interface.buttons.saveFromModule

This is a multi-cycle asynchronous operation that spans more than one PLC scan:

  1. Scan N: If RecipeHMI's program number doesn't match the active PLC recipe, loads the correct recipe from storage (via FC_RecipeStorageMap).
  2. Scan N+1: Once program numbers match, syncs PLC runtime values into RecipeHMI (preserving identity fields, same as Sync), then triggers a save to storage.

The #asyncOperationRunning flag latches the operation across scan cycles.

6. FC_RecipeStorageMap

This FC performs explicit field-by-field assignment between RecipeHMI.data and RecipeStorage.recipe[index]. It handles both directions:

  • Save (iButtonSave = TRUE): Flattens RecipeHMI.data fields into RecipeStorage.recipe[N]
  • Load (iButtonLoad = TRUE): Expands RecipeStorage.recipe[N] fields back into RecipeHMI.data

The mapping covers all 14 machine modules (header + alternating program + 64 switches + 172 setpoints). Each line is a direct 1:1 assignment. The recipe index is validated (1..100) before any operation.

7. Adding a New Recipe Parameter

When a new parameter is added to a machine module:

  1. Add the field to typeRecipeFlat UDT (naming pattern: {moduleName}_{switch|setpoint}_{fieldName})
  2. Add a Save line in FC_RecipeStorageMap: RecipeHMI.data.xxx → RecipeStorage[N].xxx
  3. Add a Load line in FC_RecipeStorageMap: RecipeStorage[N].xxx → RecipeHMI.data.xxx
  4. If the field is a header/identity field, also update the lookup arrays in RecipeHandling (Create, Edit, Delete, Save As sections)

8. Key Considerations

  • Retentive memory: RecipeStorage is RETAIN. Recipes survive power cycles.
  • RecipeHMI is NON-RETAIN: It is a temporary working buffer. Unsaved changes are lost on power loss.
  • Recipe index is 1-based: Slot 0 is not used. Input validation enforces this.
  • Button signals are expected to be pulses: Each operation runs once per button press. The HMI should send single-scan pulses.
  • Save From Module is the only multi-scan operation: All other operations complete within a single PLC scan cycle.
